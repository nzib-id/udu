## Goal

Round out the per-stat decision-making layer so every stat with a clear pre-emptive lever exposes one to the LLM, without slipping back into "answer key" prompting. Drove off two findings from the day before:

1. Wander prompt was telling the LLM to "Never pick a consume option whose expected gain is +0" — that is an inference, not a fact, so it belongs in the LLM's induction layer (or the reflection layer's rule cache), not the system prompt.
2. The annotation pattern (`expected=+Nstat`) was only on wander consume options. The feed prompt had no comparable signal, so the LLM could pick `eat_meat_raw_normal` while sickness was already 95 with no way to see the cost would clamp differently.

Also widened scope: Nzib asked whether the new pattern was applied to *every* stat with a real choice. Audit found two genuine gaps (bladder, body temperature) and two non-issues (sickness recovery + energy regen are both already mechanically handled — sickness via the SICKNESS_FUNNEL recovery rate, energy via threshold-driven sleep).

## Phase 4 design fidelity

The whole pass is bounded by the rule already documented at the top of `prompt-feed.ts`: the agent should *induce* cause→effect from its own history, not read the answer key. So:

- `expected=+N` tags = facts the LLM cannot derive otherwise (cap clamp depends on current stat). Definitional, OK to surface.
- "Skip +0" / "raw meat is risky" / "warm yourself when cold" = inferences the LLM is supposed to learn from its own observation log + reflection-cycle rules. Not in the system prompt.
- Two stats deliberately not extended:
  - **Sickness recovery** is fully passive (`SICKNESS_FUNNEL.recoveryPerGameHour = 2` while `bladder < recoveryBladderCeil` and no raw meat in system). No action to expose.
  - **Energy regen** is sleep-only and already threshold-gated. Pre-emptive sleep is not a coherent action shape (sleep is uninterruptible until `sleepWakeEnergy`).

## Day-18 bug-fix pass (afternoon → evening)

After Module 1 verify passed, ran a long autonomous session and pulled Day-18 logs. Six issues surfaced in two waves. Bug 2 (sleep⇄idle thrash) from the morning sweep folded into Bug F.

### Bug A — hunger trigger off by 0.29

`shared/config.ts` `THRESHOLDS.hungerTrigger: 25 → 35`. At `hunger=25.29` the triggered list missed by 0.29, falling through to wander LLM which picked `forage_bush` (4-tile walk + harvest cycle) instead of the in-inventory `eat_berry_inv`. Higher trigger keeps inventory-eat in the triggered loop where it belongs.

### Bug B — daily-goal `chunk_visited_new` race

`backend/src/game-loop.ts`. Daily-goal LLM takes ~9s to respond on spawn; if the char visits a new chunk during that gap the `chunk_visited_new` event arrives before `goal.status === 'in_progress'` and is dropped. Added `pendingChunkVisitedNew` flag set when event fires while goal not in_progress; replayed inside the daily-goal `.then()` callback once the goal is assigned. Reset on respawn.

### Bug C — fire-unlit recovery blocked by inventory weight

`backend/src/ai.ts` `enumerateWanderOptions`. When fire is unlit AND char can carry wood AND a known fire exists with `fuel < maxFuel/2` OR `lit === false`, surface a `pickup_wood` pre-emptive option with annotation `fire=unlit dist=Nt`. Indirect mitigation already in place via Bug A (eating berries frees inventory weight); this surfaces the lever explicitly so LLM can pick it.

### Bug D — feed-prompt JSON truncation

`backend/src/llm/choice-picker.ts` `pickWander` `numPredict: 100 → 300`. qwen3:8b consumes part of the budget on hidden `<think>` tokens even with `/no_think` suffix. Saw a real truncation at `"advances_sub` mid-key. Bumped through 200 (still truncated once) → 300 (clean). Feed-side `numPredict` left at 120 — feed prompt is shorter and never truncated.

### Bug E — drink_river spammed at high thirst

`backend/src/ai.ts` `enumerateFeedOptions`. The `+0` annotation alone wasn't enough — LLM still picked `drink_river expected=+0thirst` repeatedly. Added a hard gate: `drinkPlan = s.thirst < 60 ? planForage(...) : null`. Below 60 the option is on the menu; at/above 60 it's filtered entirely. Inventory `eat_*` options stay on the menu regardless of stat (they're the cheap recovery path).

### Bug F — sleep ⇄ idle thrash from wakeForNeed loop

`backend/src/ai.ts` + `backend/src/game-loop.ts`. Root cause: when HP ≤ `HEALTH_CONFIG.max / 2` (50), `wakeForNeed` fires every tick after the char enters sleep. Char wakes after a few seconds, energy still low, sleep re-triggers next tick, repeat. AI log shows alternating `LLM:wander_n → rule:sleep → LLM:wander_sw → rule:sleep`.

Fix: added `suppressSleepUntilTick` field on the loop. When `wakeForNeed` fires we set it to `tickCount + 6` — six game-ticks of sleep suppression to let the wake-cause need (low HP, hunger, thirst, bladder) actually be addressed before sleep re-triggers. Threaded `suppressSleep: boolean` through `decide()` signature; when true, the `sleep` need is dropped from the triggered list. Reset on respawn.

Aquarium signal: at night, the viewer sees a glowing oasis of warm light around the camp. Visually parallels the backend vision range bonus.

## Cortex mode — full LLM-driven decisions (evening pass 2)

### Goal

Eliminate the LLM⇄utility-AI handoff bugs by making the LLM the single decision-maker. Diagnosis from earlier today: every "thrash" bug observed (sleep⇄idle, drink_river spam, rest spam under terrain isolation) lives at the boundary between layers. Each layer assumes another's invariants and the assumption breaks → loops. Removing the handoff removes the bug class.

Approach (per Nzib direction): keep utility AI in the codebase as a fallback ladder, but route the primary path through a single LLM call that picks from a flat menu of every action. No mechanic disclosure in the prompt — pure Phase 4 fidelity. The LLM induces cause→effect from observation log + lessons (spirit memory across deaths), not from rules in the system prompt.

### Prompt design (Option C — hybrid Phase 4 fidelity)

- **Stat scale:** perceptual definitions only ("hunger 0=starving, 100=full"). Not strategic ("eat below 60").
- **Survival principles:** value statements only — "Health is sacred, above all stay alive", "Anticipate, don't just react", "Lessons from past lives are hard-won truths", "Survival outranks any other goal". NOT tactical hints (no "rest at fire at night", no "stockpile food").
- **Mechanic disclosure:** zero. We do not tell the LLM raw meat causes sickness, fire warms, night cuts vision. Cause→effect comes only from observation log + cached lessons.
- **`expected=+Nstat` tags** on options stay (facts about the immediate cap clamp the LLM cannot derive otherwise — same line drawn earlier today for feed/wander prompts).
- **Knowledge sources for the LLM:** stats + inventory + nearby resources (spatial only, type/id/dist) + recent observations + inherited lessons + life/daily goal. That is it.
- **Output:** strict JSON `{choice, target?, reasoning, advances_subgoal_idx, completes_subgoal}`. Same self-tag pattern as feed/wander.

### Changes

`backend/src/llm/prompt-cortex.ts` (new):
- `buildCortexPrompt(character, options, observations, rules, world, rememberedSummary, phase, gameTimeStamp)`.
- System block opens with stat scale + survival principles + sources of knowledge + output format. Reuses `formatDailyGoalBlock` from prompt-feed for the daily-plan block.
- Option formatter renders `kind=... target=... dist=Nt note=...` per option.
- Observation formatter mirrors prompt-feed exactly.

`backend/src/ai.ts`:
- New `CortexOption` type — flat shape carrying `kind | target | path | distance | annotation | finalAction`.

  - Then `enumerateWanderOptions` for direction picks + maintenance + stay (deduped by kind, e.g. `eat_berry_inv` collisions).
  - Stay maps to `rest` finalAction. Direction kinds get `idle` finalAction (game-loop installs walk_to during the path).
  - Adds unconditional `sleep` action (annotation `energy=N`) — the wander `sleep_now` is the conditional variant (night near fire); cortex always exposes the always-available `sleep`.
- `cortexDecide(...)` — same signature shape as `decide()`. Enumerates the unified menu, calls `picker.pickCortex`, builds plan from picked option. Falls back to legacy `decide()` if: picker has no `pickCortex` method, options enumerable is empty, or picker returns null (timeout / unparseable / invalid pick).

- Observation buffer cap raised **10 → 30** so cortex sees a deeper window of cause→effect history.

`backend/src/server.ts`:
- New endpoints `GET /api/admin/cortex` (returns `{enabled}`) and `POST /api/admin/cortex` (body `{enabled: bool}`). Mirrors the speed-toggle pattern.

`backend/src/llm/severity.ts` (new):

- **Patch 2 — Reasoning-first JSON schema:** schema flipped to `{"reasoning":"<one short sentence, max 15 words, grounded in current state>","choice":"<kind>","target":"<id_or_empty>","advances_subgoal_idx":<int_or_null>,"completes_subgoal":<bool_or_null>}`. Autoregressive decode forces think-before-commit.
- **Patch 6 — Severity tags:** wired through all three prompts.

`backend/src/llm/choice-picker.ts` + `backend/src/game-loop.ts` (Patch 3):
- **Patch 3 — Observation `failReason`:** new optional field on `Observation`. `executeFinal` returns string-or-undefined; every precondition fail sets a perception-level reason (not mechanic disclosure):

- `formatObservation` (cortex/feed prompts) surfaces the failReason as the action's effect tail when no stat delta fired: `tail = effects.length > 0 ? effects.join(', ') : (o.failReason ?? 'no visible effect')`.

`backend/src/llm/prompt-reflection.ts` (Patch 5):
- **Patch 5 — Anti-contamination overhaul.** SYSTEM_BLOCK rewritten:
  - Opens with "A game day has just ended" (frames the task as post-day analysis, not advice-giving).
  - Stripped positive in-prompt example (was teaching the model to mimic instead of induce).
  - 3 explicit rule shapes: `cause → effect`, `condition → consequence`, `mechanism (X works because Y)`.
  - 5 anti-pattern DON'Ts (no goals, no advice, no future-tense, no first-person, no untested generalizations).
  - Confidence anchors: 0.9+ ≥5 supporting observations, 0.7 = 3-4, 0.5 = 1-2.
  - Explicit cross-reference instruction at the end: each emitted rule MUST tag the prior it confirms/refines via `prior_idx`, or `null` if new.
- Body builder switched to numbered prior-rule list: ``priorRules.map((r, i) => `${i}. "${r.text}" (confidence ${r.confidence.toFixed(2)})`)`` — index becomes the LLM's reference handle.
- Output schema: `{"rules":[{"text":"<sentence>","confidence":<0..1>,"prior_idx":<int_or_null>}, ...],"summary":"<one short sentence>"}`.

`backend/src/rule-repo.ts` + `backend/src/llm/reflection.ts` (Patch 7):
- **Patch 7 — Confidence decay + prior_idx merge.**
- New constants in rule-repo: `CONFIRMED_BOOST = 0.05`, `UNTESTED_DECAY = 0.05`, `CONFIDENCE_FLOOR = 0.40`, `CONFIDENCE_CEIL = 0.99`.
- New `RuleRepo.replaceActive(lineageId, iteration, learnedAtDay, rules)` — wipes-and-reinserts the active set in one transaction. Old rows kept on disk (`active=0`) for audit.
- New exported `applyConfidenceDecay(priorRules, newRules)`:
  - For each new rule: `prior_idx` valid + same text → `confirmed` → boost prior conf by +0.05 (cap 0.99). Valid + different text → `refined` → take LLM's conf. `prior_idx=null` → `new` → take LLM's conf.
  - For each prior NOT referenced by any new rule → `untested` → decay -0.05. Drop below 0.40 floor.
- `reflection.ts` wired: parses `prior_idx` per rule, calls `applyConfidenceDecay`, filters below floor, calls `replaceActive`. Replaces, not appends — the merged set IS the new state.
- Rationale for params: a starting-0.55 rule dies in 3 untested days; a 0.99 rule that goes untested for a full game-week ends at 0.64 (still active). Tuning is exposed in the constants if observation shows too-aggressive decay.

`backend/src/llm/life-goal.ts` + `prompt-life-goal.ts` (Patch 4 — Cara A lineage progression, applied earlier this session):
- Diagnose+prescribe in the same LLM call. First-life path produces a self-diagnosis from the death summary; subsequent lineages inherit and refine.
- New `life_goal_diagnosis` column (migration 010). `chunks_visited_at_death` + `resources_discovered_at_death` columns added in same migration so the post-death snapshot has spatial/discovery numbers ready for the diagnose prompt.

## Phase A.1 — Tree split + auto-drop foundation

**Goal:** Replace single `tree` ResourceType with three variants (tree_fruit / tree_vine / tree_wood) and add a passive auto-drop mechanic so resources accumulate organically without forcing the LLM to shake every tree manually. Splits the food/fuel/craft economies cleanly.

### Design (agreed before implementation)

- **Tree variants:** spawn ratio 50% fruit / 25% vine / 25% wood (`TREE_TYPE_RATIO` in `shared/config.ts`). Same trunk+canopy art for all three for now.
- **Stash keys per type:** `state.fruits` (tree_fruit), `state.vines` (tree_vine), `state.branches` (tree_wood). Initialised on spawn for productive (non-barren) trees.
- **Daily regen rates:** fruit refills every 5 game-days, vine every 5 days, branch every 1 day (= 24 game-hours via `treeWoodRefillGameHours`). Caps: 2 fruit, 2 vine, 2 branch.
- **Auto-drop:** every `treeAutoDropGameHours = 12` (i.e. twice per game-day), each productive tree releases one stashed item to an adjacent free tile if any exists. Manual `shake_tree` still bypasses the cooldown for force-drop.
- **Ground items:** new ResourceTypes `branch_on_ground` and `vine_on_ground` (alongside existing `fruit_on_ground`). Pickup actions `pickup_branch` / `pickup_vine`.
- **Fire fuel:** `add_fuel` consumes `branch` first, then falls back to `wood`. Branch weight 0.8 vs wood 2.0 — branches are the post-A.1 primary fuel.
- **Fuel-economy sanity check:** at 12/7 fire schedule (12 lit hours/day × `burnPerGameHour=1`) → 12 fuel/day demand. ~13 productive tree_wood × 1 branch/day refill = balanced.
