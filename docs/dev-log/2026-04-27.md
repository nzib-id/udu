### Sleep-wake loop bug (Opsi A — minimal fix)

Nzib reported: char with low HP + low energy keeps cycling sleep → instant wake → sleep → instant wake.

**Root cause** (`game-loop.ts:1346-1350`): wake-from-sleep check included `s.health < HEALTH_CONFIG.max / 2`. When HP was low *because* of exhaustion (energy → 0 drains HP), waking was the wrong response — sleep is the only cure for exhaustion. Hunger/thirst/bladder already had their own wake clauses, so the HP clause was redundant for those causes and actively harmful for exhaustion-driven HP loss. After waking, `suppressSleepUntilTick` only delays 6 ticks; energy still low, char re-sleeps, wakes again instantly. Loop.

**Fix:** dropped `s.health < HEALTH_CONFIG.max / 2` from `wakeForNeed`. The remaining hunger/thirst/bladder triggers still wake the char if any addressable need spikes during sleep; HP-low alone (typically exhaustion or lingering sickness) no longer interrupts the recovery.

**Files:** `backend/src/game-loop.ts:1346-1349` (4 lines → 3 lines).

**Rebuild:** `docker compose up -d --build backend` — clean build, container up. Awaiting live observation that an HP-emergency + energy-low character now sleeps through to recovery instead of looping.

### Resource & Glossary Overhaul v2 — Phase 2: Continuous Coords + Server Physics (SHIPPED)

**Scope:** items get continuous float (x,y) + altitude z + velocity (vx,vy,vz). Server-side physics tick simulates gravity. Static sources (tree/bush/fire/river) keep z=0, all velocities 0. New items spawned via tree shake / hunt / manual drop carry their initial physics state.

**Design decisions locked:**
- Manual drop: exact char position, items can overlap — no offset stacking.
- Tree shake / auto-drop: use gravity (z=1.0, randomized vx/vy, vy positive = away from tree).
- Hunt: meat_raw drops at animal position on map (no longer direct to inventory).
- Server-side physics (continuous tick), not frontend tween — chosen for future throw mechanic.
- Items at rest (z=0, |v|<settleThreshold) skip physics tick.
- Pickup gate: `Math.hypot(dx,dy) ≤ 0.75` tiles.

**Physics constants (`PHYSICS_CONFIG`):**
- `gravity: 1.0` (z accel per tick — z=1 lands in ~2 ticks ≈ 1 sec at 500ms tick)
- `airFriction: 0.85` (vx/vy multiplier per tick)
- `settleThreshold: 0.05` (|v| below this and z≤0 → settled)
- `pickupRadius: 0.75` tiles

**`DROP_INIT` spawn velocities:**
- treeShake / treeAutoDrop: `{ z:1.0, vx:±0.2, vy:0.3–0.6, vz:0 }`
- hunt (drops at animal pos): `{ z:0, vx:0, vy:0, vz:0 }`
- manual drop: `{ z:0, vx:0, vy:0, vz:0 }`

**Changes:**

- **`shared/types.ts`** — `ResourceType` += `berry`, `meat_raw`, `meat_cooked` (needed for ground spawning). `Resource` type += optional physics fields `z?`, `vx?`, `vy?`, `vz?`.
- **`shared/config.ts`** — Added `PHYSICS_CONFIG` and `DROP_INIT` constants.
- **`data/migrations/011_resource_physics.sql`** — `ALTER TABLE resource_state ADD COLUMN z/vx/vy/vz REAL NOT NULL DEFAULT 0`.
- **`backend/src/resource-repo.ts`** — `persist()` writes physics columns; `rowToResource()` attaches physics fields only when non-zero (keeps WS payloads small).
- **`backend/src/game-loop.ts`**:

  - `shake` case: spawns at tree's (x,y) with `DROP_INIT.treeShake()`. Removed `findAdjacentFreeTile` path.
  - `hunt` case: drops `meat_raw` on map at animal's (x,y) with `DROP_INIT.hunt`. Removed direct inventory push + canCarry gate.

### Resource & Glossary Overhaul v2 — Phase 3: Glossary Gating + Prompt Masking (SHIPPED)

**Scope:** char no longer knows what every resource type is from spawn. Per-character `glossary` (already persisted via 012_character_glossary migration + `GlossaryRepo`) gates which actions surface to the LLM and how unknown things render in the cortex prompt. Lineage progression: gen 0 starts mostly blind; observation reveals tags; respawn inherits the parent's full glossary (already wired pre-Phase-3).

**Locked design (after Q&A iterations):**
- **Auto-known basic elements** (gen 0, pre-seeded via `seedBaseline`): `river → drinkable`, `fire → inedible`. The two unmoving anchors of survival behaviour.
- **Parent-known** (gen 0 sees parent label, subtype hidden): tree subtypes mask to `tree` until observed (`tree_fruit`, `tree_wood`, `tree_vine`); animal subtypes mask to `animal` (`animal_chicken`, `animal_fish`).
- **Full unknown** until observed: `bush` (exception — even though "tree-like", treated as opaque), all loose items (`fruit`, `berry`, `meat_raw`, `meat_cooked`, `branch`, `wood`, `vine`, `stone`, `boulder`).
- **Transformation auto-reveal:** hunt success → `meat_raw` auto-tagged; cook success → `meat_cooked` auto-tagged. Char produced these so it knows them implicitly (no observe round-trip required).
- **Observe action effects:** applies all tag effects in one shot — edible → hunger+energy, drinkable → thirst+energy, poisonous (raw meat) → sickness, inedible → no-op. Char "eats and sees" the truth.

### Phase 3 follow-up — Opsi C: generic verb kinds + opaque targets (SHIPPED + verified)

After the prompt-masking pass landed, one leak channel remained: **option `kind` strings still encoded the type they acted on** (`eat_berry_inv`, `pickup_fruit_ground`, `shake_tree_wood`, `cook_meat`, `pee_now`, `warm_at_fire`, `sleep_now`, `drink_river`). The cortex prompt would mask the inventory and observation log, then immediately list every type-coded kind right below — defeating the masking. Phase 3's design intent ("verbs are universal capability — law of nature; types are learned knowledge") forced a collapse of every action down to its bare verb, with the type/identity carried only in the `target` field (which itself gets masked via `maskType`/`maskTarget`).

**Changes:**

- **`shared/config.ts`** — Added `maskTarget(target, glossary)` helper sibling to `maskType`. Strips ALL trailing `_<digits>` groups via `/^(.+?)((?:_\d+)+)$/` (non-greedy prefix, repeated tail) so multi-segment ids like `river_38_12` correctly mask to `river_38_12` (prefix `river` is baseline-known, tail intact). The bare-prefix branch falls through to `maskType` so unknown bare types like `bush` collapse to `unknown_thing`. Single-trailing-`_\d+` regex was the initial draft and produced bugs on `river_38_12` (greedy match grabbed `river_38` as prefix → `unknown_thing_12`); the repeated-group fix is the live version.
- **`backend/src/llm/choice-picker.ts`** — `FeedKind` union collapsed to `eat | cook | pickup | shake | hunt`. `WanderKind` union collapsed to direction kinds + `stay` + `eat | drink | pickup | shake | drop | defecate | rest | sleep`. Removed `RULE_PRIORITY` const and the priority-ladder fallback in `pickFeed` (the merged kinds make priority impossible — cooked/raw/etc. all become `eat`); replaced with distance-only sort: `const sorted = [...ctx.options].sort((a, b) => a.distance - b.distance);`. `pickWander` rule fallback filters via `o.kind.startsWith('wander_') || o.kind === 'stay'` (previously matched specific direction kinds).
- **`backend/src/ai.ts`** — `enumerateFeedOptions`, `enumerateWanderOptions`, `enumerateCortexOptions` rewritten so every option emits a generic-verb `kind` and a masked `target`:
  - Feed: all `eat_<item>` → `eat` with target = inv-item type; `cook_meat` → `cook` (target masked); `pickup_fruit_ground`/`pickup_wood` → `pickup` (target masked via `maskTarget`); `forage_bush`/`shake_tree*` → `shake` (target masked); `eat_meat_raw_panic`/`eat_meat_raw_normal` collapsed into a single `eat` option with a panic flag in the annotation.
  - Wander: `eat_berry_inv`/`eat_fruit_inv` → `eat`; `drink_river` → `drink`; `pee_now` → `defecate`; `warm_at_fire` → `rest`; `sleep_now` → `sleep`; `pickup_*` → `pickup`; `shake_tree_wood`/`shake_tree_vine` → `shake`; `drop_<item>` → `drop`. Direction kinds (`wander_n` …) and `stay` unchanged.
  - Cortex enumerator dedup keys switched from bare `seenKinds` (which would now collapse all `eat`s into one) to `seenPairs` keyed on `${kind}|${target}` so distinct targets survive the merge. For wander source options, `wanderTarget = maskTarget(w.finalAction.target, g)` derives the masked label off the underlying `finalAction.target` (since `WanderOption.target` is a `Position`, not a string).
  - Observe options: `kind` is now bare `'observe'` (was `observe_${opaque}`); the opaque counter (`t1`, `t2`, …) lives in the `target` field only. LLM sees `kind=observe target=t1`.
- **`backend/src/llm/prompt-cortex.ts`** — Imported `maskTarget`. Added `formatGlossaryLine(glossary)` rendering `known: river=drinkable, fire=inedible (2 things learned)` as a dedicated line in the current-state block (after `${invLine}${fireLine}`) so the LLM has explicit visibility into what's been learned. `formatObservation` signature gained `glossary`; obs target masked via `maskTarget` before render.
- **`backend/src/llm/prompt-feed.ts`** — Same observation-masking treatment as cortex. `formatObservation(o, glossary)` masks `o.target` so even fallback feed prompt doesn't leak resource ids.
- **`backend/src/llm/prompt-wander.ts`** — SYSTEM_BLOCK rewritten: previous version still described action options using their old type-coded kind names. New version describes generic verbs only (eat/drink/pickup/shake/drop/defecate/rest/sleep), with target semantics explained per verb. `formatRecentConsume` filter: `obs.filter(o => o.action === 'eat' || o.action === 'drink')` (was a regex matching `eat_*`/`drink_*` action names).

**Verify:**
- `docker compose up -d --build backend` clean. No tsc errors after the union narrowing.
- `/api/admin/options` dump (post-rebuild, char_id=1 gen 0): cortex menu shows generic kinds only — `drop`, `drink`, `defecate`, `pickup`, `wander_*`, `stay`, `observe`. No type-coded kinds remaining.
- Live cortex pick logged: `cortex pick=drink target=river_39_7` — confirms multi-segment `maskTarget` regex is correct (river is baseline-known so prefix stays unmasked, tail `_39_7` preserved as-is). Earlier draft regex would have rendered this as `unknown_thing_7`.
- Picker validation: `parsed.choice === o.kind && parsed.target === o.target` — works unchanged because both sides agree on the new generic kind+target format.

**Open:**
- Wander LLM picker fallback path mismatches when `WanderOption.target` is a `Position` rather than a string (cortex enumerator handles this via `maskTarget(w.finalAction.target)`); acceptable degradation since cortex is primary brain.
- Phase 4 (decay) still pending — only remaining item from v2 design.

### Public-distribution hardening — admin gating (SHIPPED)

Nzib flagged the upcoming public Udu launch: viewers shouldn't be able to hit `/api/admin/kill`, toggle Cortex, or change game speed. Inventory of the surface: 12 admin endpoints (POST kill / set-stat / add-item / cortex / speed / reflect-now / daily-goal-now and GET counterparts), zero auth. Frontend exposes one of them (speed toggle) as a visible button. Fog toggle was already client-only (`localStorage`, `main.ts:151`) — no change needed.

**Backend (`server.ts`):**
- Read `process.env.ADMIN_TOKEN` once at boot. Empty → log warning and refuse all admin requests. Fail-closed.
- New `requireAdmin(req, res)` helper checks `x-admin-token` header. Mismatch → 403; missing env → 503.
- One-line guard at the top of the request handler: `if (req.url?.startsWith('/api/admin/') && !requireAdmin(req, res)) return;`. `/api/status` stays public.

**Frontend (`main.ts`):**
- `captureAdminToken()` reads `?admin=<token>` from URL, stashes in `sessionStorage` (`udu-admin-token`), then `history.replaceState` strips the param so a copy-pasted URL doesn't leak. Falls back to existing sessionStorage on subsequent loads.
- `bindSpeedToggle()` early-returns + `display:none` on `#speed-toggle` if no token. Public viewers see no admin UI at all.
- Both speed-toggle fetches now spread `adminHeaders()` (`x-admin-token: <token>` when present).

**docker-compose.yml:** backend env line `ADMIN_TOKEN: "${UDU_ADMIN_TOKEN:-}"` so compose pulls from host env or `.env`. Token generated via `openssl rand -hex 24` and written to `projects/udu/.env` (gitignored).

**scripts/observe.py:** reads `UDU_ADMIN_TOKEN` from env; new `admin_request()` wrapper sends header. Operator runs with `UDU_ADMIN_TOKEN=<token> ./scripts/observe.py`.

**Verify (smoke tests against `127.0.0.1:4246`):**

### Chicken movement + hunger system (SHIPPED)

Nzib: "ayam di map kenapa pada diem yak wkwk". Investigated, found two layered movement bugs + a missing consumption limit.

**Bug A — blocked-tile traps:** `advanceAnimals` rejected the whole step if the proposed sub-tile rounded into a blocked cell. Chickens spawning on or adjacent to blocked tiles (boulders, tree trunks, river edge) had their next step culled, target reset, and re-picked the same trap next tick. WS confirmed `chicken_0` x/y unchanged for ≥30s; DB persistence cadence (10-tick) had also masked normal motion as "stuck".

**Fix — Option A+B combo:**
- **A (sub-tile slack):** if `proposedTile === currentTile`, allow the step even though that tile is "blocked" — the chicken is already inside it, sub-tile drift can't make it more blocked.
- **B (slide-along-walls):** if proposed tile is blocked but the X-only or Y-only projection isn't, take the unblocked axis. Diagonal targets through a corner now slip past instead of stalling.
- **Zero-movement guard:** after slide, if `|nx-r.x| < 1e-6 && |ny-r.y| < 1e-6` (cardinal target → slide collapses to nothing), invoke `failHandler` to reset target/fleeing/cooldown. Stops degenerate "moving 0 px" spam.
- **River safety:** confirmed slide cannot cross water — both X-only and Y-only checks must point at unblocked tiles, and rivers are continuous strips so neighbor-tiles are blocked too.

**Bug B — fruit-chase greedy lock:** `chicken_0` still stuck after A+B. Found two failure modes:
1. Fruit landed under parent tree (blocked tile) → chicken targets it forever, slide can't reach.
2. Fruit reachable in straight line but path crosses blocked tile → slide degenerates to zero each tick.

**Fix:**
- `findNearestFruitForChicken` now skips fruits sitting on `this.blocked` tiles. Unreachable-by-design.
- New `fruitCooldownUntil` per-chicken state: when a step toward a fruit fails (slide returns zero or fully blocked), set cooldown = `tickCount + 30`. Chicken wanders normally during cooldown; fruit-chase re-enables after.

**Bug C — no consumption limit:** chickens vacuumed every fruit on the map. Nzib: "ayam ini harus dikasih limit juga biar gak ngemakanin semua resource bro".

**Fix — hunger stat (Opsi 2, future-proof):** new `ANIMAL_CONFIG` fields:
```ts
chickenHungerMax: 100,
chickenHungerStart: 60,
chickenHungerDecayPerGameHour: 2,
chickenHungerPerFruit: 30,
chickenHungerChaseThreshold: 50,
```
- Decay per tick: `(tickMs / realMsPerGameMinute / gameMinutesPerHour) * decayPerGameHour` — units consistent with Udu time convention (Nzib's correction: "udu time sama real time aja lo lupa").
- Math: 2/gh × 50 hunger = ~25 game-hours full→empty (~1 game-day). Eat threshold 50 + gain 30 → ~1 fruit per chicken per ~10 game-hours under steady state.
- `state.hunger` initialized at `chickenHungerStart` on `spawnChicken`. Decay every tick. If `hunger <= 0` → push to `starvedIds`, removed at end of `advanceAnimals` (no `meat_raw` drop — Nzib confirmed).
- Chase gate: `canChaseFruit = cooldownExpired && hunger <= chickenHungerChaseThreshold`. Wander otherwise.
- Existing `chickenRespawnGameHours: 24` loop refills count when starved chickens drop below `chickenCount`. No new respawn code needed.

**Verify (live, both chickens after restart):**
- `chicken_0` and `chicken_1` both moving (x/y change every WS broadcast).
- `chicken_0.state.hunger` decay observed: 59.40 → 59.28 over 3 sec (matches 2/gh formula).
- No starve event yet at observation time; gate passes (eat path not hit yet because hunger > 50 threshold still).
